Connect Meetily to any OpenAI-compatible API server, including self-hosted models, local servers, or third-party providers that implement the OpenAI API specification.

## Overview

The Custom OpenAI Endpoint provider allows you to:
- **Self-host models** using vLLM, LocalAI, LM Studio, or similar
- **Use alternative providers** with OpenAI-compatible APIs
- **Control costs** by running your own infrastructure
- **Ensure privacy** by keeping data on your network
- **Customize parameters** like max tokens, temperature, and top_p

**Best for:** Advanced users, self-hosted deployments, privacy-critical environments

## Setup

<Steps>
  <Step title="Start Your API Server">
    Ensure your OpenAI-compatible server is running and accessible.
    
    **Example with vLLM:**
    ```bash
    vllm serve meta-llama/Meta-Llama-3-8B-Instruct \
      --host 0.0.0.0 \
      --port 8000 \
      --api-key optional-key-here
    ```
    
    **Example with LM Studio:**
    1. Open LM Studio
    2. Load a model
    3. Click "Local Server" tab
    4. Click "Start Server" (default: http://localhost:1234)
  </Step>

  <Step title="Note Your Endpoint Details">
    Record these details from your server:
    - **Endpoint URL:** (e.g., `http://localhost:8000/v1`)
    - **Model name:** (e.g., `meta-llama/Meta-Llama-3-8B-Instruct`)
    - **API key:** (if authentication is required)
  </Step>

  <Step title="Configure in Meetily">
    1. Open **Settings > Summary** in Meetily
    2. Select **Custom Server (OpenAI)** from the Summarization Model dropdown
    3. The custom endpoint configuration form will appear
  </Step>

  <Step title="Enter Connection Details">
    Fill in the required fields:
    
    **Endpoint URL** (required):
    ```
    http://localhost:8000/v1
    ```
    
    **Model Name** (required):
    ```
    meta-llama/Meta-Llama-3-8B-Instruct
    ```
    
    **API Key** (optional):
    ```
    your-api-key-if-required
    ```
  </Step>

  <Step title="Test Connection">
    Click **Test Connection** to verify your server is reachable and the model is valid.
    
    ✓ Success: "Connection successful!"
    ✗ Failed: Review error message and check server logs
  </Step>

  <Step title="Advanced Options (Optional)">
    Click **Advanced Options** to configure:
    - **Max Tokens:** Maximum response length (e.g., 4096)
    - **Temperature:** Randomness (0.0-2.0, default: 0.7)
    - **Top P:** Nucleus sampling (0.0-1.0, default: 0.9)
  </Step>

  <Step title="Save Configuration">
    Click **Done** to save your custom endpoint configuration.
  </Step>
</Steps>

## Configuration

### Required Fields

#### Endpoint URL
The base URL of your OpenAI-compatible API server.

**Format:**
```
http://host:port/v1
```

**Examples:**
- Local server: `http://localhost:8000/v1`
- Network server: `http://192.168.1.100:8000/v1`
- Remote server: `https://api.example.com/v1`

<Warning>
  The endpoint must include the protocol (`http://` or `https://`) and typically ends with `/v1` for OpenAI compatibility.
</Warning>

#### Model Name
The exact model identifier your server expects.

**Examples:**
- vLLM: `meta-llama/Meta-Llama-3-8B-Instruct`
- LocalAI: `llama-3-8b`
- LM Studio: `local-model` (or displayed name)
- Ollama: `llama3.2:latest`

<Info>
  Model names are case-sensitive. Check your server's documentation or `/v1/models` endpoint for valid names.
</Info>

### Optional Fields

#### API Key
Authentication token if your server requires it.

**When to use:**
- Server has authentication enabled
- Remote server with security requirements
- Production deployments

**Leave empty when:**
- Local development server
- No authentication configured
- Server doesn't support API keys

#### Max Tokens
Maximum number of tokens in the response.

**Default:** Server's default (typically 2048-4096)

**Recommended values:**
- Short summaries: 1000-2000
- Standard summaries: 2000-4000
- Detailed summaries: 4000-8000

**Example:**
```
4096
```

#### Temperature
Controls randomness in responses (0.0-2.0).

**Values:**
- `0.0` - Deterministic, focused (best for factual summaries)
- `0.7` - Balanced (default, good for most use cases)
- `1.0` - More creative
- `2.0` - Highly creative (may be less accurate)

**Recommended:** `0.7` for meeting summaries

#### Top P
Nucleus sampling parameter (0.0-1.0).

**Values:**
- `0.1` - Very focused
- `0.9` - Balanced (default)
- `1.0` - Full distribution

**Recommended:** `0.9` for meeting summaries

## Testing Your Connection

The Test Connection feature verifies:
1. **Network connectivity** - Can reach the endpoint
2. **API compatibility** - Server responds to OpenAI format
3. **Model availability** - Model exists and is accessible
4. **Authentication** - API key (if provided) is valid

**Test request sent:**
```json
{
  "model": "your-model-name",
  "messages": [
    {
      "role": "user",
      "content": "Test connection. Respond with 'OK'."
    }
  ],
  "max_tokens": 10
}
```

**Success response:**
```
Connection successful!
```

**Error responses:**
- "Connection refused" - Server not running or wrong endpoint
- "Model not found" - Invalid model name
- "Unauthorized" - API key required or invalid
- "Timeout" - Server too slow to respond

### Server Configuration Examples

#### vLLM with GPU
```bash
vllm serve meta-llama/Meta-Llama-3-70B-Instruct \
  --host 0.0.0.0 \
  --port 8000 \
  --tensor-parallel-size 2 \
  --gpu-memory-utilization 0.95 \
  --max-model-len 8192
```

#### Text Generation WebUI
```bash
python server.py \
  --api \
  --extensions openai \
  --model llama-3-8b \
  --listen \
  --api-port 8000
```

#### FastChat
```bash
# Start controller
python -m fastchat.serve.controller --host 0.0.0.0

# Start model worker
python -m fastchat.serve.model_worker \
  --model-path meta-llama/Llama-3-8b-chat-hf \
  --host 0.0.0.0

# Start OpenAI API server
python -m fastchat.serve.openai_api_server \
  --host 0.0.0.0 \
  --port 8000
```

## Troubleshooting

### Connection Refused

**Error:** "Connection refused" or "Failed to connect"

**Causes:**
- Server not running
- Wrong host or port
- Firewall blocking connection

**Solution:**
1. **Verify server is running:**
   ```bash
   curl http://localhost:8000/v1/models
   ```
2. **Check endpoint URL** - Ensure protocol, host, port, and path are correct
3. **Test locally first** - Use `localhost` before trying remote servers
4. **Check firewall** - Allow connections on the server port

### Model Not Found

**Error:** "Model not found" or "Invalid model"

**Solution:**
1. **List available models:**
   ```bash
   curl http://localhost:8000/v1/models
   ```
2. **Copy exact model ID** from the response
3. **Case-sensitive** - Match the exact casing
4. **Check server logs** for model loading errors

### Authentication Errors

**Error:** "Unauthorized" or "Invalid API key"

**Solution:**
1. **Verify API key requirement** - Check if your server needs authentication
2. **Check key format** - Some servers use custom formats
3. **Leave empty** if authentication is disabled
4. **Test with curl:**
   ```bash
   curl http://localhost:8000/v1/models \
     -H "Authorization: Bearer your-api-key"
   ```

### Timeout Errors

**Error:** "Request timed out"

**Causes:**
- Server overloaded
- Model too large for hardware
- Network latency

**Solution:**
1. **Use smaller model** - Try a 7B or 8B parameter model
2. **Increase resources** - Add more GPU memory or CPU cores
3. **Reduce max tokens** - Limit response length
4. **Check server load** - Monitor CPU/GPU usage

### Invalid Response Format

**Error:** "Invalid response format" or "Parse error"

**Causes:**
- Server not fully OpenAI-compatible
- Missing /v1 in endpoint URL
- Server configuration issue

**Solution:**
1. **Add /v1 to endpoint** - Most servers require this path
2. **Check API compatibility** - Verify server supports OpenAI format
3. **Update server** - Ensure latest version with OpenAI support
4. **Test manually:**
   ```bash
   curl http://localhost:8000/v1/chat/completions \
     -H "Content-Type: application/json" \
     -d '{
       "model": "your-model",
       "messages": [{"role": "user", "content": "Test"}]
     }'
   ```

## Performance Optimization

### Hardware Requirements

**Minimum (7B-8B models):**
- CPU: 8 cores
- RAM: 16GB
- GPU: 8GB VRAM (optional but recommended)

**Recommended (13B-30B models):**
- CPU: 16 cores
- RAM: 32GB
- GPU: 24GB VRAM

**High-end (70B+ models):**
- CPU: 32+ cores
- RAM: 128GB+
- GPU: 2x A100 (80GB) or 4x A6000 (48GB)

### Optimization Tips

1. **Use GPU acceleration** - 10-100x faster than CPU
2. **Quantize models** - Use 4-bit or 8-bit quantization to reduce memory
3. **Batch requests** - Process multiple meetings together
4. **Tune KV cache** - Configure based on your context window needs
5. **Enable continuous batching** - Available in vLLM and similar servers
