# Contributing to Meetily

Thank you for your interest in contributing to Meetily! This guide will help you get started with contributing to the project.

<Info>
  Meetily is a privacy-first AI meeting assistant. When contributing, please keep privacy and local-first principles at the forefront of your work.
</Info>

## Getting Started

### Prerequisites

Before you begin, make sure you have:

- Git installed
- Development environment set up (see [Building from Source](/advanced/building-from-source))
- Familiarity with the [System Architecture](/advanced/architecture)
- A GitHub account

### Fork and Clone

1. **Fork the repository** on GitHub

2. **Clone your fork:**

```bash
git clone https://github.com/YOUR_USERNAME/meetily.git
cd meetily
```

3. **Add the original repository as upstream:**

```bash
git remote add upstream https://github.com/Zackriya-Solutions/meetily.git
```

## Development Workflow

### Branch Strategy

Meetily uses a structured branching approach:

- **`main`** - Production branch (stable releases)
- **`devtest`** - Development and testing branch
- **Feature branches** - Created from `devtest` for new work

### Creating a Feature Branch

<Steps>

<Step title="Sync with devtest">
  
  Always start from the latest `devtest` branch:
  
  ```bash
  git checkout devtest
  git pull upstream devtest
  ```
  
</Step>

<Step title="Create your branch">
  
  Use descriptive branch names:
  
  ```bash
  git checkout -b feature/your-feature-name
  # or
  git checkout -b fix/bug-description
  ```
  
</Step>

<Step title="Make your changes">
  
  Write code, update tests, and add documentation as needed.
  
</Step>

</Steps>

## Development Process

### Before You Start

<AccordionGroup>

<Accordion title="Check Existing Issues">

Before starting work:

1. Search [existing issues](https://github.com/Zackriya-Solutions/meetily/issues)
2. Check if someone is already working on it
3. Comment on the issue to let others know you're working on it

</Accordion>

<Accordion title="Create a New Issue">

If no issue exists, create one with:

- **Clear title** - Descriptive and concise
- **Detailed description** - What, why, and how
- **Steps to reproduce** (for bugs)
- **Expected behavior**
- **Screenshots** (if applicable)
- **Appropriate labels** (bug, enhancement, documentation, etc.)

**Example Bug Report:**

```markdown
## Description
Transcription stops after 5 minutes of recording

## Steps to Reproduce
1. Start a new recording
2. Let it run for 5+ minutes
3. Observe transcription stops updating

## Expected Behavior
Transcription should continue throughout the recording

## Environment
- OS: macOS 14.1
- Meetily Version: 0.1.0
- Model: whisper-small
```

</Accordion>

</AccordionGroup>

<Steps>

<Step title="Update Tests">

</Step>

<Step title="Update Documentation">
  
  - Update relevant documentation for new features

  - Document API changes
  - Add inline code comments for complex sections
  
</Step>

</Steps>

## Commit Guidelines

### Commit Message Format

Use conventional commit messages:

```
<type>(<scope>): <subject>

<body>

<footer>
```

**Types:**
- `feat` - New feature
- `fix` - Bug fix
- `docs` - Documentation changes
- `style` - Code style changes (formatting, no logic change)
- `refactor` - Code refactoring
- `test` - Adding or updating tests
- `chore` - Maintenance tasks

**Examples:**

<CodeGroup>

```git Feature
feat(audio): add support for Vulkan GPU acceleration

Implements Vulkan backend for GPU-accelerated transcription
on AMD and Intel GPUs. Includes auto-detection and fallback
to CPU if Vulkan SDK is not available.

Closes #123
```

```git Bug Fix
fix(transcription): prevent memory leak in ring buffer

Fixed issue where audio chunks were not being properly
released after transcription, causing memory growth
over long recordings.

Fixes #456
```

```git Documentation
docs(api): update transcription engine API docs

Added examples for batch transcription and improved
descriptions of model loading behavior.
```

</CodeGroup>

### Commit Best Practices

- Make small, focused commits
- Each commit should represent a logical change
- Write clear, descriptive commit messages
- Reference issue numbers when applicable

## Pull Request Process

### Creating a Pull Request

<Steps>

<Step title="Push your branch">
  
  ```bash
  git push origin feature/your-feature-name
  ```
  
</Step>

<Step title="Create PR">
  
  1. Go to the repository on GitHub
  2. Click "New Pull Request"
  3. Select `devtest` as the base branch
  4. Select your feature branch as the compare branch
  5. Fill out the PR template
  
</Step>

<Step title="Link the issue">
  
  Reference the related issue in your PR description:
  
  ```markdown
  Fixes #123
  Closes #456
  Related to #789
  ```
  
</Step>

</Steps>

### Pull Request Template

Use this template for your PRs:

```markdown
## Description
[Describe your changes here]

## Related Issue
Fixes #[issue number]

## Type of Change
- [ ] Bug fix
- [ ] New feature
- [ ] Documentation update
- [ ] Performance improvement
- [ ] Code refactoring
- [ ] Other (please describe)

## Testing
- [ ] Unit tests added/updated
- [ ] Manual testing performed
- [ ] All tests pass

## Documentation
- [ ] Documentation updated
- [ ] No documentation needed

## Checklist
- [ ] Code follows project style
- [ ] Self-reviewed the code
- [ ] Added comments for complex code
- [ ] Updated README if needed
- [ ] No breaking changes (or documented)
```

### Review Process

<AccordionGroup>

<Accordion title="What to Expect">

1. **Initial Review** - A maintainer will review within 2-3 days
2. **CI Checks** - Automated tests must pass
3. **Code Review** - At least one maintainer approval required
4. **Revisions** - Address any feedback or requested changes
5. **Merge** - Once approved, maintainers will merge to `devtest`

</Accordion>

<Accordion title="Responding to Feedback">

- Be open to suggestions and constructive criticism
- Ask questions if feedback is unclear
- Make requested changes promptly
- Keep the PR up to date with `devtest`
- Respond to all comments before requesting re-review

</Accordion>

<Accordion title="Keeping Your PR Updated">

If `devtest` has moved ahead:

```bash
git checkout devtest
git pull upstream devtest
git checkout feature/your-feature-name
git rebase devtest
git push --force-with-lease origin feature/your-feature-name
```

</Accordion>

</AccordionGroup>

## Code Style

### Rust Code

- Follow Rust standard formatting (`cargo fmt`)
- Use `clippy` for linting (`cargo clippy`)
- Prefer explicit types for public APIs
- Use meaningful variable names
- Add doc comments for public functions

**Example:**

```rust
/// Starts audio recording with the specified devices.
///
/// # Arguments
/// * `mic_device_name` - Optional name of the microphone device
/// * `system_device_name` - Optional name of the system audio device
///
/// # Returns
/// Result containing () on success or error message on failure
#[tauri::command]
pub async fn start_recording(
    mic_device_name: Option<String>,
    system_device_name: Option<String>,
) -> Result<(), String> {
    // Implementation
}
```

### TypeScript/JavaScript Code

- Use TypeScript for type safety
- Follow ESLint configuration
- Use async/await over promises
- Prefer functional components (React)
- Use meaningful component and variable names

**Example:**

```typescript
import { invoke } from '@tauri-apps/api/core';

interface RecordingOptions {
  micDeviceName?: string;
  systemDeviceName?: string;
  meetingName: string;
}

export async function startRecording(
  options: RecordingOptions
): Promise<void> {
  try {
    await invoke('start_recording', {
      mic_device_name: options.micDeviceName,
      system_device_name: options.systemDeviceName,
      meeting_name: options.meetingName,
    });
  } catch (error) {
    console.error('Failed to start recording:', error);
    throw error;
  }
}
```
